# Phase 03 Completion Evidence — Service Layer Implementation

## Verdict: PASS

## Summary

Replaced the three global streaming fields on `ChatServiceImpl`
(`is_streaming: AtomicBool`, `current_conversation_id: Option<Uuid>`,
`stream_task: Option<JoinHandle<()>>`) with a per-conversation map
`active_streams: Arc<StdMutex<HashMap<Uuid, ActiveStream>>>`, where
`ActiveStream { task: JoinHandle<()>, cancel: CancellationToken }`.

`begin_stream(conversation_id)` now only rejects if that specific
conversation already has an active entry; other conversations are
unaffected. `cancel_active_stream(conversation_id)` removes, cancels, and
aborts only the targeted entry, and emits `ChatEvent::StreamCancelled`
with the same `conversation_id`. `spawn_stream_task` inserts the new
entry keyed by `conversation_id` without touching other entries. The
`ChatService` trait now carries `cancel(&self, conversation_id: Uuid)`
and `is_streaming_for(&self, conversation_id: Uuid)`; all callers
(presenter, tests, coverage harnesses, E2E) were updated.

Approval resolution on cancel remains `resolve_all(false)` — scoping it
to a single conversation is explicitly deferred to Phase 07.

## Semantic Verification

Traced cancel flow end-to-end in code:
- `ChatService::cancel(&self, Uuid)` → `cancel_active_stream(Uuid)`.
- `cancel_active_stream` locks `active_streams`, removes the entry for
  the provided id; if present, calls `entry.cancel.cancel()` and
  `entry.task.abort()`; unconditionally calls
  `approval_gate.resolve_all(false)`, pushes
  `ViewCommand::ToolApprovalResolved`, and emits
  `ChatEvent::StreamCancelled { conversation_id, .. }`.
- Other conversations remain untouched in the map, so their tasks
  continue running.

Test `cancel_scopes_to_conversation` exercises this: two mock streams
are inserted (A and B), cancel is called for A only, then the map is
inspected — only B remains. Test
`cancel_emits_event_only_for_target` asserts the event stream sees
exactly one `StreamCancelled` for A and zero for B.
