## Named Startup-Atomicity Proof Artifact

### Startup hydration mode classification
**Mode A** — transcript outcome is synchronously available. `build_startup_inputs()` calls `conversation_service.get_messages()` and includes the result in `StartupInputs.selected_conversation.mode = StartupMode::ModeA { transcript_result }`.

### Exact shared startup/runtime selection-loading entry path
Both startup and runtime use:
- `begin_selection_locked(&mut inner, conversation_id, mode)` (L331 startup / L275 runtime in `app_store.rs`)
- `reduce_view_command_without_publish(inner, ConversationMessagesLoaded { conversation_id, selection_generation, messages })` (L344 startup / L283+ runtime)

### Startup atomicity proof

**Exact test**: `startup_and_manual_selection_converge_on_one_authoritative_delivery_path` in `tests/presenter_selection_and_settings_tests.rs`

**Command**: `cargo test --test presenter_selection_and_settings_tests -- startup_and_manual --nocapture`

**Observed result**: PASS. Store state after `from_startup_inputs()` shows:
- `selected_conversation_id = Some(conv_id)`
- `transcript = [2 messages]`
- `load_state = Ready { conversation_id, generation: 1 }`
- `selection_generation = 1`

**Why this proves the required startup ordering**:
1. `from_startup_inputs(inputs)` creates `AppStoreInner::default()` (no subscribers)
2. `reduce_startup_batch(&mut inner, inputs)` runs the full batch synchronously: profiles → conversations → `begin_selection_locked` → `reduce_view_command_without_publish(ConversationMessagesLoaded)` → sets `revision = 1`
3. Only after the batch completes does `Self { inner: Mutex::new(inner) }` construct the store
4. No `subscribe()` or `current_snapshot()` is possible before construction completes
5. The first consumer reads the already-committed `Ready { generation: 1 }` snapshot

**Why this falsifies any pre-hydration-default implementation**: `AppStoreInner::default()` starts with `load_state = Idle`, `selection_generation = 0`, empty transcript. If a subscriber could observe before `reduce_startup_batch` completes, it would see `Idle`/empty. But the struct doesn't exist for subscription before the batch finishes.

**Why this falsifies visible Loading for known transcript**: `begin_selection_locked` sets `load_state = Loading` inside the batch, but `ConversationMessagesLoaded` immediately transitions to `Ready` in the same batch. No publication occurs between these steps because `from_startup_inputs` uses `BatchNoPublish` mode and the store struct hasn't been constructed yet.
