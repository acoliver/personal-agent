## Requirements Implemented (Expanded)

### REQ-ARCH-001: Authoritative App Store

**Full Text**: GPUI runtime MUST expose one authoritative store for chat-facing application state.

**Behavior**:
- GIVEN: startup hydration and runtime presenter updates
- WHEN: both mutate state
- THEN: one durable store owns the selected conversation, transcript snapshot, loading state, and related snapshots used by popup views

**Why This Matters**: This is the core convergence point that removes popup-coupled correctness.

### REQ-ARCH-003: Explicit Conversation Loading Protocol

**Full Text**: Selection must be an explicit loading protocol/state, not “clear and hope replay arrives”.

**Behavior**:
- GIVEN: a manual or startup conversation selection
- WHEN: activation happens before transcript replay
- THEN: the store enters explicit loading state and preserves coherent visible state until replacement is ready

**Why This Matters**: Prevents false empty-transcript renders and stale replay races.

## Structural Checks

- All four required Phase 02 design artifacts exist at the expected plan paths.
- `analysis/02-authoritative-store-contract.md` is structured as an implementation contract, not freeform notes, with explicit sections for ownership, loading protocol, selection-generation dispatch, startup transaction contract, protocol migration, compatibility boundaries, and preservation guarantees.
- `analysis/pseudocode/01-app-store.md` and `analysis/pseudocode/02-selection-loading-protocol.md` are numbered, reducer-oriented, and implementation-usable.
- `analysis/pseudocode/03-main-panel-integration.md` ties the store contract to the app-root runtime seam, popup subscription model, and sole production drainer target.

## Semantic Verification

- Singular authority is explicit: the contract assigns durable ownership of selected conversation id/title, transcript snapshot, loading state, generation, and popup-facing snapshots to one authoritative store rooted in app-root GPUI runtime state, while explicitly disallowing popup-local durable ownership, second generation counters, second transcript mirrors, and startup-specific semantic mutators outside the canonical startup transaction.
- Loading protocol is explicit and non-aspirational: the contract names `Idle`, `Loading`, `Ready`, and `Error`, specifies generation matching rules, requires stale payload drops, states that `ConversationActivated` is non-authoritative/idempotent in ordinary runtime, and preserves previous transcript state instead of clearing and hoping replay arrives.
- No-clear ordinary activation is explicit: both contract and pseudocode state that ordinary `ConversationActivated` does not clear transcript, and `ConversationMessagesLoaded` remains bulk replacement only for matching conversation id and generation.
- Generation ownership is singular and implementation-usable: `handle_select_conversation_intent(...)` plus `begin_selection(...)` are defined as the only ordinary-runtime selection issuance path and sole minting site, with explicit same-id no-op vs retry semantics and explicit synchronous failure handling when `GpuiBridge::emit(...)` fails.
- Startup/runtime convergence is explicit: the contract requires startup hydration to use the same reducer helper path through `reduce_startup_batch(startup_inputs)` plus `begin_selection(..., BatchNoPublish)` and matching success/failure completion, with bounded Mode B only when transcript outcome is genuinely unavailable.
- Popup independence is explicit: the contract and integration pseudocode move draining responsibility to app-root `spawn_runtime_bridge_pump(...)`, require it to survive zero mounted popups, and define popup reopen as immediate snapshot subscription rather than replay coupling.
- Preservation constraints are carried forward: bulk replacement, no-clear activation, exact finalize-stream durability, bounded `ConversationCleared` compatibility, and popup-non-authority rules are all explicitly retained.
- Repo grounding remains intact: the contract cites current seams in `src/main_gpui.rs`, `src/ui_gpui/views/main_panel.rs`, `src/ui_gpui/bridge/gpui_bridge.rs`, `src/ui_gpui/views/chat_view.rs`, and `src/presentation/view_command.rs`, and it explicitly treats current `ViewCommand` shapes as migration targets rather than assuming already-landed implementation.

## Evidence Summary

- The validated artifacts already satisfy the phase requirements without needing semantic rewrites.
- The docs are concrete enough for later implementation phases to rely on without guessing: they define the target store module, the sole ordinary-runtime selection minting boundary, the startup transaction boundary, explicit reducer behavior for `ConversationActivated`, `ConversationMessagesLoaded`, `ConversationLoadFailed`, `FinalizeStream`, and `ConversationCleared`, and the popup-independent runtime bridge pump target.
- Repo checks confirm the current code still has the pre-recovery seams the contract is designed to remediate: current `ViewCommand` variants lack generation on `ConversationActivated`/`ConversationMessagesLoaded`, `ensure_bridge_polling(...)` remains popup-bound, `build_startup_view_commands(...)` plus `apply_startup_commands(...)` still exist, and current views/presenter still use popup-coupled selection and replay paths.
- Because the contract explicitly closes alternate semantic owners and defines exact authority boundaries, later implementation phases can implement against it without ambiguity.
