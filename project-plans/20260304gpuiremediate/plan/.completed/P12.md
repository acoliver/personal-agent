## Final Architecture Invariants

1. One authoritative store (`GpuiAppStore` at `src/ui_gpui/app_store.rs`) owns chat/history/settings state for the GPUI process lifetime.
2. Startup and runtime transcript delivery use the same reducer path (`reduce_view_command_without_publish`).
3. Popup open/close does not affect store state delivery or correctness.
4. `begin_selection` is the sole ordinary-runtime `selection_generation` minting site.
5. `spawn_runtime_bridge_pump` is the sole production `drain_commands` owner.
6. Views render from store snapshots and emit intents only via `selection_intent_channel`.
7. `FinalizeStream` uses direct-finalize durable model with dedupe guard.
8. `ConversationCleared` stays bounded non-authoritative; store revision/transcript unchanged.
9. Same-id reselection from Loading/Ready is a no-op; from Error retries with new generation.
10. Stale/off-target commands do not bump revision or publish.

## Remaining Compatibility Shims or Follow-Ups

1. **Settings/Profile initial data**: Settings views get initial data through `request_runtime_snapshots()` → presenter responses via bridge, not through store snapshot. The settings domain is in the store snapshot but not yet consumed by settings views directly.
2. **`ConversationCleared` same-turn render readback**: ChatView clears local state on `ConversationCleared` but does not yet call `apply_store_snapshot(current_snapshot())` to restore from authoritative snapshot. The store itself is unaffected (proven by test). A follow-up could add explicit snapshot restoration.
3. **`ensure_bridge_polling`**: Retained as documented no-op for structural compatibility. Can be removed in cleanup.
4. **Pre-existing clippy debt**: 1018 clippy warnings in unrelated files. Zero in recovery-touched files.
5. **`ShowToolCall`/`UpdateToolCall`**: Non-rendered transport passthrough. No GPUI rendering exists. Unchanged by recovery.
