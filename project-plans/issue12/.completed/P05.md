# Phase 05 Evidence — Integration, Migration Mapping, and Regression Guards

## Verdict: PASS

## Summary

Phase 05 delivers the legacy settings migration mapping and startup-restore
regression guards specified in the plan:

**`src/ui_gpui/theme.rs`** — New public function `migrate_legacy_theme_slug(slug: &str) -> &str`
placed in a dedicated `// ── Legacy slug migration ──` section.  Maps:
- `"dark"`  → `"default"`       (old name for the bundled dark theme)
- `"light"` → `"default-light"` (old name for the bundled light theme)
- `"auto"`  → `"mac-native"`    (old name for the OS-appearance mode)

All other values pass through unchanged.  The function is idempotent: applying
it twice on the output of the first call returns the same result.

**`src/main_gpui/startup.rs`** — `build_startup_inputs_async` updated to:
1. Import `migrate_legacy_theme_slug` from the library crate.
2. Apply `migrate_legacy_theme_slug` to the raw slug read from
   `AppSettingsService::get_theme()` before calling `set_active_theme_slug`.
3. Log both the raw persisted slug and the migrated slug for diagnostics.

**`tests/theme_migration_startup_tests.rs`** — New integration test file with
20 tests covering:
- Unit: all three legacy mappings (`dark`, `light`, `auto`).
- Unit: passthrough for all canonical slugs and unknown values.
- Unit: idempotency of migration for all three legacy values.
- Startup-restore: all three legacy startup paths + canonical + no-slug-persisted.
- Round-trip: `select → persist → migrate → restore` cycle for canonical slugs.
- Round-trip: all bundled + synthetic slugs survive migration unchanged.
- Regression: legacy slugs must NOT appear in `available_theme_options`.
- Regression: migrated legacy slugs MUST appear in `available_theme_options`.
