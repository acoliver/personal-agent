8. Keyboard & Mouse Behavior
Navigation
Esc / Cmd+W  →  Back to Chat
Tab  →  Cycle text input fields in active panel
Within Content Panels
↑ / ↓  →  Navigate list items (profiles, MCPs)
Enter  →  Confirm / submit
+  →  Add new item (context-dependent)
E  →  Edit selected item
Mouse / Scroll
Click sidebar category → switch panel
Mouse wheel → scrolls only the active panel's list
One scrollable element per panel — no nested scroll conflict
Key change from current: arrow keys only affect the active panel's list,
not multiple lists simultaneously (current behavior scrolls profiles and themes together).
9. GPUI Implementation Notes
Layout Structure
div().flex().flex_col().size_full() // settings-view .child(render_top_bar()) // 44px fixed .child( div().flex_1().flex().flex_row() // settings-body .child(render_category_sidebar()) // 120px fixed .child(render_content_panel()) // flex-1 )
New State
enum SettingsCategory { General, Models, Security, McpTools, } // Add to SettingsState: // selected_category: SettingsCategory // theme_dropdown_open: bool
Theme Dropdown
Standard dropdown like conv-dropdown / profile-pill:
• Click trigger → toggle theme_dropdown_open
• Click item → emit SelectTheme, close dropdown
• Click elsewhere → close dropdown
No Outer Scroll
Content panel uses overflow_hidden().
Only the list box or dropdown list gets overflow_y_scroll().
One scrollable element per screen, max.
SettingsView stays one GPUI entity. Render switches on selected_category.
All existing events, bridge, and state management stay structurally the same.
Profile/MCP sub-views continue using navigation_channel for full-screen navigation.
